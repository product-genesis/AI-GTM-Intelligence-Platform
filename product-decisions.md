
**Issue 1** -  Users experienced noticeable latency when navigating between pages caused by repeated backend calls for static account and GTM configuration data. 
    
Action: Worked with the frontend developer to design a local‑storage caching strategy for semi‑static data. On initial sign‑up, data is fetched once from the backend and stored in local Storage. On subsequent navigation, the UI reads that data from the browser cache, eliminating network round‑trips for these fields. This drastically reduced perceived latency after first‑time load and during navigation.

**Issue 2** — Signals generated by the platform were not relevant to the actual company information using the product. They reflected a generic, fictional company profile instead of the real one.

Observations
A live signal referenced "North America/Europe" as target markets. The actual company profile in Mission Control did not contain this. The signals looked coherent and well-formed but were describing the wrong company.

Investigation
Traced the signal generation flow from the frontend service call (signals.ts) through to the backend batch generation endpoint. Reviewed what data was being passed in the request body when POST /api/generate-signals-batch_claude was called.

Root Cause
The frontend service was sending hardcoded placeholder values (generic industry, size, geography) to the backend instead of reading from the real company profile stored in the system. The LLM received fictional input and generated accurate signals — for a company that did not exist.

Recommendations
Discussed with the Engineering team and they suggested to replace hardcoded placeholders in signals.ts with live values read from the company profile, using safe fallbacks where fields may be empty. The system must not generate signals if the company profile is incomplete — surface a prompt to complete it first.

Learnings
An LLM producing confident, coherent output is not evidence the input was correct. Grounding failures are silent — the output looks valid until you cross-reference it against reality. AI pipelines need input validation, not just output quality checks.

**Issue 3** - The Profiler agent was not using any of the uploaded data sources (PDFs, documents indexed in Pinecone) when generating signals. It was drawing on its training data instead, producing plausible but fabricated reasoning.

Observations
Profiler-generated signals referenced strategies and context that did not appear in any uploaded document. The Scout agent, by contrast, did receive some Pinecone results — but without any label or description of what those results represented.

Investigation
Reviewed the backend batch generation service (batch.py). Traced how Pinecone query results were retrieved and passed into the LLM context for both Scout and Profiler. Examined how context_json was constructed before being sent to the model.

Root Cause
Two related failures in the same pipeline:

Scout: Pinecone results were appended to the context blob without a label. The LLM received the content but had no description of what it was — customer intelligence, market data, internal documents — making correct attribution unreliable.
Profiler: Pinecone results were retrieved and stored in a local variable (profiler_pre_data) but then excluded when context_json was assembled. Only company_profile and icp_data were included. The data source context was silently dropped before the LLM ever saw it.

Learnings
Missing context is more dangerous than wrong context. Wrong context produces off-target output; missing context produces hallucination that is harder to detect because the LLM fills the gap with confident, plausible fabrication. 

**Other decisions made** - 

1. In contrast to the earlier version, I introduced business logic and additional refinement to **Signals** and **Reports** so they stay at a 100-foot view, and remain directly relevant to business decision-making.

2. Made the AI context-aware and intelligence-led so that every signal and report directly connects to relevant prospects. This move ensured every insight leads to real opportunities.

3. Introduced clear reasoning behind every recommendation made by Agentic AI, so users understand:
	- Why this lead matters (fit, intent, and potential impact)
	- Why now is the right time (timing, urgency, and buying window)
	- What signals triggered this (events, behaviors, and data points)

4. Added agentic conversation layer so users don’t have to jump between dashboards. Instead, they can talk to the system through context-aware conversations grounded in their data, market signals, and current workflow. The platform can then reason in real time, surface what matters, and guide the next best action.
